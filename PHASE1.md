# Phase 1 Plan

Build the playable core first: a Mafia round with real game logic, real NPC agent calls, visible state, and enough voice to prove the concept. Do not start by building the full Path D architecture from `BUILD.md`. Path D is the full product version; Phase 1 is the demo-stable version that proves the loop.

Phase 1 must not be a static storyboard or a fake chat mock. The game engine owns roles, phases, legal actions, votes, eliminations, and win/loss. The NPCs are generated by an LLM from the current game state. The player can participate as Player 6.

## Goal

By the end of Phase 1, a player should be able to sit down, receive a secret role, hear or read five AI NPCs accuse each other, speak or type as Player 6, vote, see an elimination, and understand why the project is unusual.

The target experience:

> A candlelit Mafia table where five AI characters argue in distinct voices, the human player participates, and the game advances through night, day discussion, vote, elimination, and win/loss.

## Non-Goals

These are valuable, but not Phase 1 blockers:

- Full ElevenLabs Agents Platform orchestration
- Custom LLM SSE endpoint connected to an ElevenLabs Agent
- Server tools for every game action
- SQLite-backed event replay
- Three.js postprocessing and particle polish
- Audio-reactive shader glow
- Fully autonomous real-time turn-taking
- Perfect standard Mafia depth
- Cloud deployment

If these are attempted too early, they can consume the whole build and leave no playable demo.

## Stack Decision

Use a single Next.js app for Phase 1.

Recommended stack:

- Next.js App Router
- TypeScript
- Plain CSS in `app/globals.css`
- Server Route Handlers for game and voice endpoints
- In-memory demo sessions, optionally serialized to `localStorage` from the browser
- OpenAI Node SDK for NPC generation
- ElevenLabs REST TTS when keys are present
- Browser speech synthesis fallback when ElevenLabs is unavailable

Do not start with a separate Bun/Hono backend. It is the right shape for Path D, but it adds process coordination before the demo loop exists. Phase 1 should keep server state and UI in one local app, then split later only if the ElevenLabs Agent integration requires it.

Use Tailwind only if the scaffold already has it working quickly. Plain CSS is acceptable and faster to debug while the core loop is still changing.

## Phase 1 Architecture

Use the simplest architecture that can support the playable loop.

```text
Next.js app
  - Game UI
  - API-backed game state
  - Role card
  - Transcript
  - Vote grid
  - Human input: speech if ready, text fallback always
  - 2D noir table scene first

Next.js API routes
  - Start game
  - Advance phase
  - Generate NPC speech
  - Submit human speech
  - Submit votes/night actions
  - Return canonical game state

OpenAI
  - Generate NPC dialogue and simple decisions from current state

ElevenLabs
  - Direct TTS per speaker first
  - Multi-voice Agent path only after core loop is stable
```

The browser should never invent game facts. It renders state and submits player actions. The server-side game module validates and advances the game.

## Build Order

### 1. Scaffold The Runnable App

Create a minimal app that runs locally.

Required output:

- `package.json`
- Next.js app
- Basic route at `/`
- Environment loading
- One command to run the demo

Keep this boring. Do not spend time on monorepo perfection.

### 2. Implement The Real Game Engine

Implement a compact 6-player Mafia state machine.

Players:

- Don Vito
- Salvatore
- Rosa
- Vincenzo
- Carmela
- Player 6

Roles:

- 2 Mafia
- 1 Detective
- 1 Doctor
- 2 Villagers

Required state:

- player id
- display name
- alive/dead
- role
- phase
- day number
- current speaker
- public transcript
- private NPC inner monologues
- votes
- night actions
- winner

Required phases:

- setup
- role-reveal
- night
- day-discussion
- day-vote
- resolve-vote
- game-over

Phase 1 can simplify standard Mafia:

- Use one compressed night action set per night.
- If both Mafia are NPCs, let one LLM or deterministic helper choose the kill.
- If human has a night role, show a target picker.
- Resolve Doctor save before kill.
- Detective result can appear privately in UI if human is Detective.

Engine functions to implement:

- `createGame(seed?)`
- `assignRoles(players, seed?)`
- `getLegalTargets(state, actorId, action)`
- `submitHumanSpeech(state, text)`
- `generateNpcTurn(state, npcId)`
- `submitVote(state, voterId, targetId)`
- `submitNightAction(state, actorId, targetId)`
- `resolveNight(state)`
- `resolveVote(state)`
- `checkWinCondition(state)`
- `advancePhase(state)`

The reducer should return a new state object and transcript events. Avoid scattered mutation in UI components.

### 3. Make Talking To Agents Work

Generate short character turns with OpenAI.

Required output per NPC turn:

```json
{
  "inner_monologue": "short hidden thought",
  "speech": "public line of dialogue",
  "vote": null,
  "role_action": null
}
```

For Phase 1, do not require XML voice tags in the model response. Generate clean text, then wrap or route it in code.

Prompt inputs:

- persona
- private role
- current phase
- alive players
- public transcript summary
- private knowledge
- current objective

Keep each NPC line short. Demo pacing matters more than rich simulation.

Required behavior:

- NPC day-discussion turns call OpenAI.
- NPC votes call OpenAI or a deterministic fallback using suspicion.
- NPC night actions call OpenAI or a deterministic fallback using legal targets.
- If OpenAI fails, each NPC still emits an in-character fallback line and a legal action.

This means the game remains playable offline-ish, but when the API key works, the player is genuinely talking to LLM-driven characters.

### 4. Build The API Contract

Use server routes as the boundary between UI and engine.

Suggested routes:

- `POST /api/game/start`
- `GET /api/game/:id`
- `POST /api/game/:id/human-speech`
- `POST /api/game/:id/human-night-action`
- `POST /api/game/:id/human-vote`
- `POST /api/game/:id/advance`
- `POST /api/speak`

The `advance` route should perform the next server-owned step:

- narrator line
- NPC night action
- resolve night
- next NPC discussion turn
- NPC vote
- resolve vote
- game-over check

This gives the UI one reliable "continue" button during development and allows automatic pacing later.

### 5. Human Turn

Always support typed input.

Speech capture is out of scope. Do not add mic capture, browser speech recognition, Scribe STT, or push-to-talk controls.

Required UI:

- typed text field
- submit button
- visible transcript entry as `Player 6`

Typed input keeps the game playable without any mic permission flow.

### 6. Voting

Implement click-to-vote first.

Required behavior:

- show living players only
- prevent self-vote if desired, but self-vote can be allowed for simplicity
- NPC votes generated with constrained valid target list
- tally visibly
- eliminate highest vote-getter
- deterministic tie rule

Recommended tie rule:

> If tied, eliminate the tied player with the highest suspicion count from NPC votes. If still tied, choose the first tied player in seating order.

This is not perfect Mafia, but it prevents demo deadlocks.

### 7. Visual Demo Surface

Start with a 2D scene, not Three.js.

Required visuals:

- dark Palermo/noir table background
- five NPC portraits or silhouettes
- one Player 6 position
- candle or center light
- active speaker glow
- dead player dimmed or crossed out
- phase banner
- role card
- transcript
- vote grid

Use CSS effects first:

- `image-rendering: pixelated`
- box-shadow glow for active speaker
- brightness drop during night
- red flash on elimination

Three.js is Phase 2 unless the 2D demo is already stable.

### 8. Voice Layer

Use direct ElevenLabs TTS first.

Required behavior:

- map each character to a voice id
- when an NPC speaks, play that speaker's TTS audio
- show transcript even if audio fails
- allow mute/skip

Phase 1 voice hierarchy:

1. Direct TTS per character
2. Direct TTS only for narrator + current speaker
3. Browser speech synthesis fallback
4. Text-only fallback

Do not block the game loop on perfect multi-voice Agent integration.

### 9. SFX and Music

Add cheap atmosphere after the loop works.

Required sounds:

- low background loop
- night transition sting
- vote/elimination sound

Generated ElevenLabs SFX are good for the pitch, but local static files are acceptable for Phase 1 if generation slows the build.

### 10. Pre-Staged Scenarios

Create at least two deterministic seeds.

Scenario A:

- Don Vito is Mafia
- Salvatore is innocent but suspicious
- Day 1 creates a strong accusation scene

Scenario B:

- Carmela is Mafia
- Vincenzo accidentally accuses correctly
- Rosa over-explains and becomes a false suspect

The local demo should be able to restart into a known good run.

## Overnight Implementation Milestones

### Milestone A: Text Game

Deliver:

- runnable app
- start game
- role card
- server-owned state
- phase banner
- transcript
- human typed speech
- deterministic NPC fallback lines
- voting and elimination

No voice yet. No OpenAI dependency yet. This proves the game exists.

### Milestone B: Agent Game

Deliver:

- OpenAI-backed NPC discussion
- OpenAI-backed NPC vote/night target selection
- persona-specific prompts
- fallback generation when OpenAI fails
- compressed but real round flow

This proves talking to agents works.

### Milestone C: Voice Game

Deliver:

- `/api/speak`
- ElevenLabs REST TTS if configured
- browser speech synthesis fallback
- per-character voice mapping
- visible transcript always
- skip/mute controls

This proves voice is part of the loop without betting everything on the full Agent platform.

### Milestone D: Polished Local Game

Deliver:

- polished 2D noir table
- active speaker state
- dead/alive state
- night/day lighting
- elimination flash
- deterministic demo seeds
- one full rehearsed run

This is the Phase 1 finish line.

## Critical Gates

### Gate 1: Playable Text Loop

Deadline: first major checkpoint.

Pass condition:

- start game
- assign roles
- advance night/day/vote
- human can submit input
- NPCs speak
- vote resolves
- game can reach win/loss

If this is not working, cut voice and visual complexity.

### Gate 2: Agent Proof

Pass condition:

- at least three NPC turns are generated by OpenAI from the actual game state
- the human's submitted text appears in later NPC prompt context
- NPC votes and night actions resolve to legal targets
- fallback generation keeps the game moving if OpenAI fails

If this is not working, stop UI polish and fix the agent/game boundary.

### Gate 3: Voice Proof

Pass condition:

- at least three distinct NPC voices can play from game-generated lines
- transcript remains visible
- game does not freeze when TTS fails

If this is not working, use narrator voice plus text transcript.

### Gate 4: Demo Surface

Pass condition:

- player can understand who is alive, who is speaking, what phase it is, and what they can do next
- role card is obvious
- voting is obvious
- elimination is dramatic

If this is not working, stop adding backend features and polish the UI.

### Gate 5: Demo Rehearsal

Pass condition:

- one full 4-5 minute run works twice in a row
- restart works
- fallback path is known
- audio can be muted or skipped

After this gate, only fix bugs and polish visible moments.

## Implementation Notes

### Keep State Canonical

All UI should render from one canonical `GameState`. Avoid hidden state in components for game facts like alive/dead, phase, votes, or role.

### Keep AI Bounded

The LLM should not own the game engine. It only chooses:

- what an NPC says
- optional vote target
- optional night target
- inner monologue

Code owns:

- phase transitions
- legal targets
- vote validity
- eliminations
- win condition

### Keep Prompts Short

Long prompts will slow the live demo. Summarize transcript and private knowledge aggressively.

A good NPC turn should be one or two sentences.

### Keep Fallbacks Visible

Every voice action must have text on screen. Every human input must be typed. Every generated action must have a deterministic fallback.

## Definition of Done

Phase 1 is done when:

- A new game can start from the browser.
- Player 6 receives a private role.
- NPCs generate character-specific dialogue.
- Human input affects the transcript.
- A vote can be cast and resolved.
- A player can be eliminated.
- The game can declare town or Mafia win.
- At least one voice path works.
- The UI has a noir table, active speaker state, role card, transcript, and voting controls.
- A deterministic demo seed can be loaded.

At that point, the project is demoable. Phase 2 can then pursue the full Path D upgrade: ElevenLabs Agent, Custom LLM SSE, multi-voice XML routing, server tools, Three.js POV, and richer replay.

## Phase 2 Upgrade Path

After Phase 1 works, upgrade in this order:

1. Replace direct TTS with multi-voice XML routing.
2. Add Custom LLM SSE endpoint for ElevenLabs Agent.
3. Move game actions into server tools.
5. Split backend into Bun/Hono if the Agent integration needs a dedicated service.
6. Replace 2D scene with Three.js POV.
7. Add post-game inner monologue replay.

Do not begin this list until the Phase 1 definition of done is met.
